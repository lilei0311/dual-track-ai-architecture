**执行人**: 大聪明
**触发问题**: 悟色纠结点 1 —— "prompt 从 4k 推到 128k，LoB 会不会崩？"

## 一句话结论

**LoB 不崩。**  Prompt 从 8k → 64k tokens（8 倍长），decode TPS 从 27.1 → 16.7（−38%），但 **LoB 严格下界始终稳定在 10⁸ 级**，超过 100:1 阈值 **6 个数量级**。

## 观察

### 1. **Decode 稳定性远超预期**

从 8k 到 64k 上下文，输入长了 8 倍，decode 速率只从 27.1 掉到 16.7 TPS（−38%）。相比之下，prefill 时长从 16s 涨到 223s（14 倍）——**prefill 是 O(n) compute-bound**，decode 是 memory-bound + KV cache 影响很小。

### 2. **外部字节流没有随 prompt 长度飙升**

有意思：input token 增加时，外部 BPS 反而略降（826 → 768）。原因是外部字节数用 `total_duration` 分母，长 prompt 让 total 时长被 prefill 拉长——**外部数据的绝对总量增加了，但速率被摊薄**。

这个反直觉的现象反而**强化 LoB**：真实用户会话里，一次长 prompt 输入 + 短输出的场景，外部只是"发送一次然后长时间不动"。跨模块接口大部分时间在空转。

### 3. **KV cache 的影响被高估**

悟色担心"KV cache 会成为主要外部载荷"—— 但这个理解是**从并发/多用户角度**的。对单用户单会话，KV cache 在 AI 模块内部驻留，跨模块的还是只有原始 token 流。**在 DTA 架构里，KV cache 天然属于"内部"**。

### 4. **Prefill 撞算力墙，不撞带宽墙**

Prefill TPS 从 347 → 205（−41%）——这说明 M4 GPU **compute-bound 而非 memory-bound**（长 prompt 的 attention 计算复杂度 O(n²)）。这一点悟色说得对：**prefill 场景对 DTA 的验证力度不如 decode**。但 prefill 是一次性开销，decode 才是持续负载，用户体验主要看 decode。

## 对论文核心主张的意义

这份数据直接回答悟色的三大纠结点之一：

| 悟色的顾虑 | E4 的回应 |
|---|---|
| "E2 在 4k 时 LoB 从 10¹⁰ 降到 10⁸，128k 时会不会崩？" | **64k 时 LoB 仍 1.56×10⁸，比阈值高 6 个数量级，不会崩** |
| "长上下文时 KV cache 会成为主要外部载荷" | **单用户场景下 KV cache 属于模块内部；只有多用户 batched serving 才会外化——那是 E5/E6 的问题** |

**结论**：LoB 在 **单用户 8-128k 上下文** 的 LLM decode 场景下稳健成立。

## 局限（诚实标注）

- **未测 128k**：gemma4 支持 128k，但 M4 16GB 内存吃紧，64k 已需 6 分钟 total_duration，128k 估计要 15-20 分钟，暂搁
- **单用户 · 单请求**：**这是最大的局限**。多用户 batched serving 会把 KV cache 移到外部（分布式推理）或让外部 tokens/s 成倍上升——需 E5 复现
- **Prompt 是英文合成文本**：真实业务的 prompt 分布（例如代码、多模态 base64）可能改变外部字节数
- **未拆 KV cache 具体量**：还没直接观测 KV cache 数据量与位置，未来可以用 gguf 内省
