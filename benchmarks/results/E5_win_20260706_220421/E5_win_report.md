## 2. 测试方法

使用 `benchmarks/E5_win_run.ps1 -Model qwen2.5:3b`，通过本地 Ollama HTTP API 运行 4 个场景：

| 场景 | prompt 说明 | max_tokens |
|---|---|---|
| E5_1_baseline | 500 词内存带宽解释 | 800 |
| E5_2_prompt2k | "The quick brown fox..." × 100 | 200 |
| E5_3_prompt6k | "The quick brown fox..." × 500 | 200 |
| E5_4_prompt8k | "Once upon a time..." × 800 | 100 |

LoB（Logic-over-Bytes）严格下界：

```
LoB_strict = GPU 峰值显存带宽 (GB/s × 1e9) / 外部总字节流 (bytes/s)
外部总字节流 = (prompt_bytes + response_bytes) / wall_elapsed_sec
```

LoB 宽松上界使用模型权重扫描估算（params × tokens / elapsed）。

## 4. 关键观察

1. **同段位带宽、不同拓扑**：RTX 2050（112 GB/s）与 M4 base（120 GB/s）带宽几乎同级，但前者是独立 GDDR6，后者是 UMA 共享内存。如果 M4 侧 `qwen2.5:3b` 数据出来后 LoB 同数量级，将直接支持“LoB 是带宽结构性质，与内存拓扑无关”。

2. **LoB 数量级**：排除 baseline 冷启动异常后，prompt2k/6k/8k 的严格 LoB 落在 **3.8×10⁷ ~ 2.0×10⁸** 区间，全部远高于 100:1；prompt2k 甚至超过 10⁸。

3. **decode TPS 偏低**：RTX 2050 上 `qwen2.5:3b` 的 wall decode TPS 仅 2–13 tok/s，远低于 M4 侧 gemma4:9b 的 29.2 tok/s。这很可能是 RTX 2050 算力（2048 CUDA cores）和 4 GB 显存瓶颈导致，不影响 LoB 结论，但说明该模型在该卡上更接近“带宽/算力双受限”。

4. **baseline 建议复跑**：若需要与 M4 侧 E1 baseline 严格对齐，可在模型已驻留显存后复跑一次 baseline，用真实 decode 时长替换冷启动 wall 时间。
